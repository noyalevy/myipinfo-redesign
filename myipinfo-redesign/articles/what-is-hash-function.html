What Is a Hash Function? SHA-256, MD5 & More Explained

📅 March 2025⏱ 7 min read✍️ MyIPInfo Team

A hash function takes any input — a word, a file, an entire database — and produces a fixed-length "fingerprint." Change one character and the fingerprint changes completely. This simple concept powers passwords, blockchain, digital signatures, and more.

What Is a Hash Function?

A hash function is a mathematical algorithm that takes an input of any size and produces an output — called a hash, digest, or checksum — of a fixed length. No matter whether you feed it a single letter or a 10 GB video file, the output is always the same length.

SHA-256("hello") = 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824
SHA-256("hello!") = ce06092fb948d9af698b3a74e636a9626a7c5bfc563e6c31f3b5a27f6b0d7ed4

Notice: adding just one character (!) completely changes the hash. This is called the avalanche effect — a core property of good hash functions.

Key Properties of Cryptographic Hashes

  • Deterministic — The same input always produces the same hash
  • Fast to compute — Hashing is quick even for large inputs
  • One-way (pre-image resistant) — You cannot reverse a hash back to the original input
  • Avalanche effect — A tiny input change produces a completely different hash
  • Collision resistant — It should be computationally infeasible to find two different inputs that produce the same hash
💡 Hash vs. Encryption

Hashing is one-way — you cannot decrypt a hash. Encryption is two-way — you can decrypt ciphertext with the right key. Never confuse the two: passwords should be hashed, not encrypted.

Hash Algorithms: MD5, SHA-1, SHA-256, SHA-512

MD5 (Message Digest 5) — 128-bit

MD5 was designed in 1992 and produces a 32-character hex hash. It was once widely used for passwords and file integrity, but is now considered cryptographically broken. Researchers demonstrated collision attacks in 2004 — meaning two different files can produce the same MD5 hash. Do not use MD5 for security. It's only acceptable for non-security checksums (verifying a download wasn't corrupted).

SHA-1 (Secure Hash Algorithm 1) — 160-bit

SHA-1 produces a 40-character hex hash. Used heavily through the 2000s in TLS certificates and Git commit IDs. Google's SHAttered attack (2017) demonstrated a practical SHA-1 collision. SHA-1 is deprecated for security use — browsers no longer accept SHA-1 TLS certificates.

SHA-256 — 256-bit ✅ Recommended

SHA-256 is part of the SHA-2 family, producing a 64-character hex hash. It is the current gold standard for most security applications. Used in Bitcoin mining, TLS 1.3 certificates, JWT signatures, and AWS request signing. No practical attacks exist against SHA-256.

SHA-512 — 512-bit

SHA-512 produces a 128-character hex hash — double the size of SHA-256. It provides extra security margin and can actually be faster than SHA-256 on 64-bit processors due to its design. Preferred for high-security applications like government and financial systems.

SHA-3 — The Next Generation

SHA-3 uses a completely different internal structure (Keccak sponge construction) from SHA-2. It was standardized in 2015 as a backup in case weaknesses are found in SHA-2. Available in 224, 256, 384, and 512-bit variants.

Real-World Use Cases

  • Password storage — Websites store password hashes (with salt), not the passwords themselves. When you log in, your password is hashed and compared to the stored hash.
  • File integrity — Download pages often provide a SHA-256 hash so you can verify the file wasn't corrupted or tampered with.
  • Blockchain / Bitcoin — Bitcoin uses SHA-256 to link blocks together and for proof-of-work mining.
  • Digital signatures — Documents are hashed, and the hash is signed — much faster than signing the full document.
  • Git version control — Every commit in Git is identified by a SHA-1 hash (migrating to SHA-256).
  • Deduplication — Cloud storage services hash files to identify duplicates without reading content.

Frequently Asked Questions

Can a hash be reversed? +
No. Cryptographic hash functions are one-way — you cannot mathematically reverse a hash to get the original input. However, attackers can use precomputed "rainbow tables" of common inputs and their hashes. This is why passwords should be hashed with a unique salt.
What is a hash collision? +
A collision occurs when two different inputs produce the same hash output. In theory, collisions must exist (infinite inputs, finite outputs). A secure hash function makes collisions computationally infeasible to find. MD5 and SHA-1 have known practical collision attacks; SHA-256 does not.
What is salting a hash? +
Salting adds a random unique value (the "salt") to a password before hashing. This prevents rainbow table attacks — even if two users have the same password, their salted hashes will be completely different. Modern password libraries like bcrypt, Argon2, and scrypt handle salting automatically.
Which hash algorithm should I use? +
For most security uses: SHA-256. For passwords specifically, use a dedicated password hashing algorithm like bcrypt, scrypt, or Argon2 — these are designed to be slow, making brute-force attacks harder. Never use MD5 or SHA-1 for security-sensitive applications.
